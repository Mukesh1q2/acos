## 1. Which File Types Should Always Be Committed Immediately

### Tier 1 — Commit on creation (zero tolerance for loss)

| File Type | Why | Example |
|-----------|-----|---------|
| **Source code** (`.py`, `.ts`, `.tsx`) | Cannot reproduce without it | `stiefel.py`, `models.py`, `losses.py` |
| **Experiment configs** (`.yaml`, `.json`, `.toml`) | Define what was run | `config.json`, `hyperparams.yaml` |
| **Requirements/lock files** | Define the environment | `requirements.txt`, `pyproject.toml`, `bun.lock` |
| **Database schema** | Define data structure | `prisma/schema.prisma`, SQL migrations |
| **Experiment result JSONs** | The primary evidence | `results/experiment_a.json` |
| **Markdown reports** | The interpreted evidence | `AFM_EXPERIMENT_REPORT.md` |
| **Random seed logs** | Required for exact reproduction | `seeds.log`, `rng_states/` |

### Tier 2 — Commit after each experiment run

| File Type | Why | Example |
|-----------|-----|---------|
| **Trained model weights** (`.pt`, `.pth`) | Needed for evaluation/resumption | `checkpoints/afm_epoch50.pt` |
| **Training logs** (`.csv`, `.log`) | Per-epoch metrics | `logs/afm_train.csv` |
| **Evaluation outputs** | Reproduce figures/tables | `eval/confusion_matrix.npy` |
| **Statistical test outputs** | Cannot recompute without raw data | `results/statistical_tests.json` |

### Tier 3 — Commit at project milestones

| File Type | Why | Example |
|-----------|-----|---------|
| **Large databases** (>10MB) | Grow over time, expensive to regenerate | `data/scientific_validation.db` |
| **Generated figures** | Reproduce paper/report content | `figures/kl_collapse.png` |
| **Environment snapshots** | Record exact package versions | `pip_freeze.txt` |

### The rule:

> **If a file cannot be regenerated from other committed files, it is Tier 1.**  
> **If regenerating it requires a GPU/long compute, it is Tier 2.**  
> **If it can be regenerated from Tier 1+2 files, it is Tier 3.**

The afm-lite loss occurred because ALL result files were Tier 1/2 but were never committed.

## 2. Which Files Can Safely Remain Temporary

| File Type | Why Temporary | Regeneration Method |
|-----------|---------------|-------------------|
| `__pycache__/` | Auto-regenerated by Python | Delete and re-import |
| `.next/` | Auto-regenerated by Next.js | `bun run dev` |
| `node_modules/` | Regenerated from `package.json` | `bun install` |
| `.pt` checkpoints (if JSON results exist) | Results contain the important numbers | Re-train from code |
| `dev.log` | Runtime artifact | Auto-created |
| `/tmp/` contents | Ephemeral by design | N/A |
| Screenshots in `download/` | QA artifacts, not evidence | Re-capture |
| `agent-ctx/*.md` | Agent working notes | Re-generate |

### The boundary:

A file is safe to leave temporary **only if** either:
1. It is auto-regenerated by tooling (`node_modules`, `__pycache__`), OR
2. It can be fully reproduced from other committed files (training logs from code + config)

A file is NOT safe to leave temporary if:
- It contains unique observational data (experiment results)
- It requires expensive compute to regenerate (trained models)
- It records a specific random event (logs, traces)

## 7. Checkpoint vs JSON vs Report: Separate Commits

**Yes. They should be committed separately, with distinct messages.**

### Rationale

Each artifact type answers a different question:

| Artifact | Answers | Failure Mode if Mixed |
|----------|---------|----------------------|
| **Source code** | "What was run?" | Code changes hidden in a commit about results |
| **JSON results** | "What were the numbers?" | Cannot tell if results changed due to code change or re-run |
| **Reports** | "What do the numbers mean?" | Interpretation changes attributed to data changes |
| **Model checkpoints** | "Can we resume/evaluate?" | Large binary blobs make git history hard to browse |
| **Environment records** | "Can we reproduce the setup?" | Dependency changes invisible if bundled with code commits |

### Commit strategy

```
commit 1: "afm-lite: add source code (stiefel.py, models.py, losses.py, data.py, train.py, experiments.py, run_all.py)"
commit 2: "afm-lite: add requirements.txt and pyproject.toml"  
commit 3: "afm-lite: add v0.1 experiment results (experiment_a/c/d/e.json, exp_b_vae.json, multi_run_stats.json, statistical_tests.json)"
commit 4: "afm-lite: add AFM_EXPERIMENT_REPORT.md"
commit 5: "afm-lite: add pip_freeze_container.txt (environment record)"
commit 6: "afm-lite: add SHA256 manifest for integrity verification"
tag:     "afm-v0.1-restored"
```

### For model checkpoints specifically

Model weights (`.pt` files) are typically 10-100MB each. Git handles them poorly. Options:

1. **Don't commit checkpoints** — if you have the code + config + seeds, you can re-train
2. **Commit only the final checkpoint** — one per experiment, not per-epoch
3. **Use Git LFS** — if available, store large files separately
4. **Store externally** — upload to a file host and record the URL in a manifest

**Recommendation for AFM:** Don't commit checkpoints. The 602K-param model trains on MNIST in minutes on CPU. The important artifacts are the code, config, seeds, and JSON results.
